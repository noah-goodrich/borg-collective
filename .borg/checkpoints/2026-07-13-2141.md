## 3. The blocker — why PR #33 isn't merged

**claude-plugins CI (`hook BATS suites + shellcheck`) has been RED on `main` since #30 (2026-07-09)
and #31** — both merged over failure. PR #33 inherits that red. The failing tests are in
`borg-collective/hooks/test/borg-link-down.bats`:

- `not ok 12  project mode: output (if any) is valid JSON`
- `not ok 14  project mode: no plan-mode nudge when PROJECT_PLAN.md exists`
- `not ok 15  project mode: uncommitted reminder appears when state.json has has_uncommitted_changes=true`

All three fail with `jq: parse error: Invalid numeric literal at line 1, column 88`. This is a
**JSON-assembly bug in `borg-link-down.sh`'s project-mode `additionalContext` output** when
plan/uncommitted context fixtures are present — the emitted JSON is malformed.

Key facts for diagnosis:
- The guard change is **orthogonal** — it's an early-exit *before* any JSON output. Running the built
  hook locally with a valid project cwd emits **valid JSON**; the failure only reproduces under the
  bats fixtures (state.json with `has_uncommitted_changes=true`, PROJECT_PLAN.md present).
- `scripts/build-plugin.sh` does **not** sync test files; `borg-collective/hooks/test/borg-link-down.bats`
  is maintained inside the **claude-plugins** repo (274 lines), not in borg-collective.
- Likely cause: the `additionalContext` string is concatenated into the JSON without proper `jq`
  escaping when the extra context pieces (plan nudge / uncommitted reminder) are appended. Find where
  the hook builds the final `{"hookSpecificOutput":{...,"additionalContext": ...}}` and ensure the
  whole context string is assembled via `jq --arg` / `jq -Rs`, not printf interpolation.
- Fix the source in **borg-collective `hooks/borg-link-down.sh`**, add/adjust a regression test, then
  re-run `scripts/build-plugin.sh` to propagate into claude-plugins.
